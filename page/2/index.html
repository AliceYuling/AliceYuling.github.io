web安全之CSRF攻击

什么是 CSRF

跨站请求伪造(Cross-Site Request Forery), 通常缩写为 CSRF 或 XSRF, 是指一种恶意的网站应用，通过伪装成受信任用户的请求来利用受信任的网站完成攻击。

CSRF 基本流程

  • 用户登录 A 网站 a.com
  • 用户在保持 A 网站登陆状态的同时，受引访问了 B 网站 b.com
  • B 网站向 A 网站发送请求，如 a.com?action=xxx, 其请求中往往带有一些有损 A 网站用户或者窃取信息等的操作
  • A 网站在收到 B 网站发来的请求后，此时由于A 网站用户的登录凭证还保留在 cookie中，A 网站默认使用受害用户的 cookie
  • A 网站以受害用户的名义执行了攻击者请求中的操作，CSRF 攻击完成

下图描述了 CSRF 的基本流程(图源见水印)

如何防范 CSRF 攻击

设置 Referer header

CSRF 攻击只能通过冒用他人 cookie 而不能读取 cookie。针对该特点，可以通过同源检测来防范 CSRF 攻击。

Http 请求头中有一个名为 referer 的字段，这个字段的作用提供访问来源，即告诉服务器当前请求是从哪个页面链接过来的。利用 referer, 服务器可以判断发起请求的来源，阻止不可信的外域请求。

但是这种方法也并不是完美的，它具有以下缺点：

  • Referer 在各浏览器的实现不尽相同，难以保证浏览器本身没有安全漏洞

  • 用户可能自己设置浏览器使其在发送请求时不带 referer，这种情况下，当他们访问网站时，网站会因为其没有 referer 而认为是 CSRF 攻击，从而阻挡掉一部分正常用户的请求。

请求时附加额外信息进行验证

同样的利用CSRF 攻击只能通过冒用他人 cookie 而不能读取用户信息如 cookie 的特点，CSRF 攻击主要是因为服务器把攻击者的请求当成了是安全用户的请求。那么如果要求用户请求时带上一些额外的验证信息，如 token，那么服务器在接收到请求时，便可以通过判断 token 是否有效来验证请求是否安全。关于 token 认证具体原理与流程可参考上一篇博客。

基于Token的身份验证方案

客户端在向服务端请求资源的时候，由于有些资源是需要用户身份才能拿到的，这时候就要进行用户身份认证。而HTTP协议是无状态的，即当一个客户端请求发送给服务器的时候，服务器并不知道上一次发送请求的客户端的身份。每次发送请求都必须重新登录进行身份，要解决这个问题，常见的方案有基于会话的身份验证方案和基于Token的身份验证方案。

传统身份验证的方法

传统的用户身份认证基于会话。用户请求登录时，服务端经过验证登录信息没有问题时，会生成一条会话记录，这条记录里面说明了登录的用户是谁。服务器端返回给客户端一个 sessionID，客户端将 sessionID 存储在 Cookie 中，在下次这个用户请求时，带上这个Cookie，服务器端收到这个Cookie后，取出其中的 sessionID 并在已存储的会话信息中查找，如果找到了该sessionID的话，说明用户已登录过尚在会话期间，那么就将用户请求的数据返回而无需重新登录。

基于会话的用户身份认证存在以下的问题：

  • 对每个用户的认证信息都需要存储在服务端，当用户会话增多之后，会增加服务器的存储开销
  • 容易受到XSRF攻击

基于 Token 的身份认证方式

基于 Token 的身份认证方式主要工作流程如下：

  1. 用户携带用户名和密码等信息请求访问

  2. 服务端校验用户信息没有问题后，生成一个 token

  3. 服务端将 token 返回给客户端

  4. 客户端将 token 存储在本地

  5. 在之后的每一次请求中客户端都携带 token， 服务端校验 token

Token 的实施 – JWT

JWT 的全程为 JSON Web Tokens，它是一个一个开放标准(RFC 7519)，它定义了一种紧凑的、自包含的方式，用于作为JSON对象在各方之间安全地传输信息。

JWT 标准的 Token 有三个部分：

  • header
  • payload
  • signature

这三部分中间用 . 分隔开。

Header 是 JWT 的头部数据，里面包含了使用的算法与 token 的类型的 JSON 对象。如：

将此JSON对象使用 base64 算法转换为字符串即为 JWT header。

Payload 是 JWT 的主体内容部分， 也是一个 JSON 对象，其中包含了需要传递的数据，如发行人、到期时间、发布时间等。

除了以上七个 JWT 定义的默认的字段外，还可自定义字段。 同样使用 base64 算法转换成字符串后即为 JWT 的 payload 部分。

Signature 是签名部分，对 header 和 payload 通过指定的算法加密后生成，在加密时指定一个 密钥（secret）， 该密钥存储在服务器中不得向用户公开。

以上三部分分别计算出后，使用 . 号分隔组合成的字符串即为 JWT token。

如何防止 token 冒充

客户端在接收到服务端返回的 token 后，一般把 token 存在浏览器 local storage 或者其他存储中，那么如果有人复制了这个 token，拷贝到另一个客户端使用，该怎么防止这种情况呢？

为了防止 token 冒充，服务端在生成 token 时使用 UserAgent 作为干扰码，不同的客户端干扰码不一样，那么直接将 token 拷贝到另一个客户端使用则无法验证成功，在一定程度上起到了防冒充的作用。

深度优先算法和广度优先算法实现深拷贝

深拷贝&浅拷贝

对于 js 中的简单数据类型，浅拷贝就对值的复制，而对于对象来说，浅拷贝只复制一层对象的属性，并不包括对象里面的为引用类型的数据。修改其中一个对象的属性，则另一个对象的属性也会改变。与赋值不同的是，把一个对象赋值给另一个对象，二者指向同一对象；而把一个对象浅拷贝给另一个对象，二者并不指向同一对象。

深拷贝则是开辟新的栈，两个对象对应两个不同的地址，修改一个对象的属性，不会改变另一个对象的属性。

深度优先和广度优先

深度优先的思想即对数据结构的每个子节点一层一层深度遍历结束后再进行下一个子节点的深度遍历。而广度优先则是从数据结构的最外层遍历，将同一层的子节点遍历完后再进行下一层子节点的遍历。

深度优先一种实现方式是在取得要遍历的节点时如果发现有子节点（children） 时，则不断的深度遍历，并把这些节点放入一个待搜索的栈（Stack）中，直到最后一个没有子节点的节点时，开始对栈进行搜索。后进先出（下列代码中使用了 push 方法入栈，因此需使用 pop 方法出栈），如果没有匹配到，则删掉此节点，同时删掉父节点中的自身，不断重复遍历直到匹配为止。

而广度优先的实现则可先把外层的数据结构放入一个待搜索的队列（Queue）中，进而对这个队列进行遍历，当正在遍历的节点存在子节点（children）时则把此子节点下所有节点放入待搜索队列的末端。

Service Worker学习小结

什么是 Service Worker

Service Worker 是一段运行在浏览器后台中的脚本，它独立于当前页面，提供了那些不需要与web页面交互的功能在网页背后悄悄执行的能力。 [1] Service Worker 是 PWA 当中的一部分，它是一个可编程的网络代理，可以控制 Web 页面的所有网络请求。 Service Worker 独立于 JavaScript 主进程，它无法直接访问 DOM 和 window 对象。Service Worker具有以下作用：

  • 浏览器缓存
  • 消息推送
  • 实现离线 Web App

Service Worker 具有独立的生命周期， 如下图所示：

Service Worker 具有下载、安装、激活三个生命周期。

  • 下载： 用户首次访问Service Worker控制的网站时，Serivce Worker会被下载

  • 安装：下载完成后，开始安装阶段，通常在安装阶段进行一些资源的加载和缓存

  • 激活： 安装完成后，开始激活 Service Worker，这个阶段可以对之前缓存的资源的管理。激活之后，Service Worker 即开始对其所属范围内的页面进行控制。

docker基础学习

Docker

Dockers是是一个开源的应用容器引擎，让开发者将应用以及依赖包打包到一个可移植的容器中，然后发布到任何流行的 Linux或Windows 机器上，也可以实现虚拟化。

docker 镜像

Docker 镜像是一个特殊的文件系统，除了提供容器运行时所需的程序、库、资源、配置等文件外，还包含了一些为运行时准备的一些配置参数（如匿名卷、环境变量、用户等）。镜像不包含任何动态数据，其内容在构建之后也不会被改变。

docker 容器

容器就是将软件打包成标准化单元，以用于开发、交付和部署，是镜像运行的实体。
容器和虚机的区别：传统虚拟机技术是虚拟出一套硬件后，在其上运行一个完整操作系统，在该系统上再运行所需应用进程；而容器内的应用进程直接运行于宿主的内核，容器内没有自己的内核，而且也没有进行硬件虚拟。因此容器要比传统虚拟机更为轻便。

仓库

Docker Registry 是一个集中的存储、分发镜像的服务。通常，一个仓库会包含同一个软件不同版本的镜像，而标签就常用于对应该软件的各个版本。我们可以通过 <仓库名>:<标签> 的格式来指定具体是这个软件哪个版本的镜像。如果不给出标签，将以 latest 作为默认标签。

Docker Registry 公开服务是开放给用户使用、允许用户管理镜像的 Registry 服务。一般这类公开服务允许用户免费上传、下载公开的镜像，并可能提供收费服务供用户管理私有镜像。最常使用的 Registry 公开服务是官方的 Docker Hub，这也是默认的 Registry，并拥有大量的高质量的官方镜像。除此以外，还有 CoreOS 的 Quay.io，CoreOS 相关的镜像存储在这里；Google 的 Google Container Registry，Kubernetes 的镜像使用的就是这个服务。

节流和防抖函数

前两天写轮播图的时候，发现了一个问题就是在快速多次点击翻页按钮的时候，会多次触发翻页函数，此时由于上一次翻页的动画还没结束，就会出现很多问题。这个时候就需要用到节流函数。节流和防抖这两个概念经常都是一起出现的，自己对这两个概念和方法也不是很熟，在这篇博客里就放到一起写了。

节流/防抖的概念及区别

节流(throttle)： 某个函数在一个特定的时间内只执行一次。比如调整窗口大小时，resize事件会持续被触发，使用节流函数可以让其在事件触发期间每间隔一段时间执行一次，而不是每触发一次事件执行一次事件函数。

防抖（debounce)： 在事件触发后一段时间t内没检测到事件时则开始执行事件函数，若在t内再次检测到了事件触发那么重新开始计时。也就是防抖是在一个动作结束后才执行。

下面这张图可以很形象的说明节流和防抖的区别，第一行是正常的事件触发频率，可以看到防抖是把在动作结束后一段时间内未再次检测到事件触发则执行，它可以把函数的多次调用压缩成一次。而节流则相当于减满了函数的执行速度。

图片来自 (http://demo.nimius.net/debounce_throttle/)

以下可能会高频触发的事件可能会导致页面卡顿、浏览器奔溃等情况出现，如果在事件函数中写了异步操作的话，可能会导致结果和预想的不一样，为了绕开这些问题，可以使用节流/防抖：

  • click
  • resize
  • input
  • scroll
  • mousemove
  • …

节流比防抖更适合用于一些触发更频繁的场景中，如动画相关的resize/scroll等，而防抖适合用于input/keyup

节流的实现

事件触发时，计算这次触发的时间与上次执行函数的时间之差，如果时间差达到设定的时间间隔，则执行这个函数，否则不执行。

防抖的实现

使用定时器，在每次事件触发时，清除定时器，直到设定的时间内没有事件触发，则函数被执行
